\section{Introducción}
Trasplan es una plataforma web y móvil diseñada para coordinar y gestionar el proceso de procuramiento y trasplante de órganos en tiempo real. Facilita la comunicación entre los centros médicos, los médicos de la Unidad de Cuidados Intensivos (UCI), los coordinadores de trasplantes y los procuradores para optimizar la detección, evaluación, activación familiar, asignación y cirugía de órganos de donantes potenciales.

\section{Arquitectura y Estructura del Proyecto}
El repositorio está dividido en cuatro componentes principales:
\begin{enumerate}
    \item \textbf{backend/}: Servidor de base de datos y API GraphQL (Node.js + Express + Apollo Server v4 + Sequelize + TypeScript).
    \item \textbf{web/}: Panel administrativo y operacional web (React + Material-UI v4 + Apollo Client v3 + TypeScript).
    \item \textbf{App/}: Aplicación móvil nativa moderna (React Native CLI + TypeScript).
    \item \textbf{mobile/}: Versión móvil antigua/legada basada en Expo (Expo SDK 37 + JavaScript).
\end{enumerate}

\section{Historial de Desarrollo}
El desarrollo de la plataforma Trasplan ha seguido un proceso interdisciplinario y colaborativo entre las Facultades de Medicina e Ingeniería de la Pontificia Universidad Católica de Chile:
\begin{itemize}
    \item \textbf{Etapa I (2020) --- Concurso Interdisciplina VRI UC}: Conceptualización inicial y codiseño del proyecto liderado por el Dr. Briceño, Dr. Achurra, Dr. Neyem y S. Riveros.

    \item \textbf{Etapa IV (2022) --- Postulaciones FONIS}: Postulación a fondos concursables FONIS 2022 y FONIS 2023 (no adjudicados).
    \item \textbf{Etapa V (2023--2024) --- Consolidación}: Finalización del desarrollo informático y estabilización de la plataforma.

\section{Dominio de Negocio y Flujo de Procuramiento}
El sistema modela el ciclo de vida completo de un potencial donante de órganos, desde la sospecha inicial hasta el trasplante definitivo. El diseño de Trasplan aborda de forma directa los problemas históricos detectados en el proceso de procuramiento en Chile:
\begin{itemize}
    \item \textbf{Fase de Detección}: Retraso u omisión en la notificación. Para mitigar esto, según la Ley de notificación obligatoria de potencial donante, el sistema permite que cualquier médico inicie una alerta de \textbf{Sospecha} en terreno al detectar un paciente con daño neurológico severo y puntaje de Glasgow $<$ 8.
    \item \textbf{Fase de Mantenimiento}: Falta de coordinación de equipos de procura, logística de traslado y control del tiempo de isquemia fría. Trasplan centraliza la información para eliminar la ``información limitada'' durante la cirugía de procura y optimiza la comunicación.
    \item \textbf{Fase de Trasplante}: Pérdida de órganos viables debido a expiración de tiempos de isquemia o ineficiencias en la asignación.
\end{itemize}

\subsection{Módulos y Modelos de Datos}

\begin{temariobox}{Resumen de Módulos Clínicos y Operacionales}
\begin{itemize}
    \item \textbf{User / Workplace / MedicalCenter}: Gestión de usuarios (médicos, coordinadores, procuradores), sus lugares de trabajo y los centros de salud asociados.
    \item \textbf{Suspicion (Sospecha)}: Registro inicial de un donante potencial en el hospital de origen (identificado por su RUT único y sexo). Habilitado por el botón de pánico en la app móvil.
    \item \textbf{InclusionCriteria}: Cuestionario para verificar si el paciente cumple los requisitos médicos y legales para ser donante (descarte de infecciones como VIH, VHB, VHC, o condiciones como Sepsis y Cáncer).
    \item \textbf{MedicalRecord / Background}: Ficha médica con características físicas básicas (peso, estatura, grupo sanguíneo) y antecedentes de salud (enfermedades crónicas, infecciones, hábitos).
    \item \textbf{BrainDeath}: Registro de pruebas clínicas (como test de apnea, reflejos del tronco encefálico y electroencefalograma) para verificar formalmente la muerte cerebral por parte de 2 médicos (uno de ellos neurólogo/neurocirujano).
    \item \textbf{FamilyActivation / Confirmation / Negation}: Registro de la entrevista familiar, capturando el consentimiento (detallando qué órganos se autoriza donar) o la denegación (con motivos como motivos religiosos, culturales, etc.).
    \item \textbf{Organ / OrganType}: Órganos del donante aptos para procuramiento (corazón, hígado, riñones, etc.), con su tiempo de isquemia específico.
    \item \textbf{Offering (Oferta)}: Oferta de un órgano a centros médicos receptores compatibles de manera secuencial y priorizada.
    \item \textbf{Surgery (Cirugía)}: Planificación de la extracción en pabellón, coordinando el equipo quirúrgico y la hora crítica de clampeo.
    \item \textbf{HistoryItem / ExpoDeviceToken}: Auditoría de cada acción ejecutada en las sospechas y tokens para notificaciones push en dispositivos móviles.
    \item \textbf{Statistics}: Módulo de analítica que proporciona visualizaciones y métricas regionales demográficas, rendimiento e indicadores de donantes y traslados por centro asistencial.
    \item \textbf{ClinicalSupport \& EducationalTools}: Herramientas complementarias para el personal clínico que incluyen guías clínicas de mantención de donantes (preservación en UCI), educación interactiva sobre donación para profesionales y guías de apoyo familiar para entrevistas sensibles.
\end{itemize}
\end{temariobox}

\section{Ciclo de Vida y Transiciones de Estado}
Tanto las sospechas de donantes (Suspicion) como los órganos (Organ) y las ofertas (Offering) poseen flujos de estado rigurosos:

\subsection{1. Estados de una Sospecha (Suspicion)}
\begin{itemize}
    \item \textbf{Sospecha}: Registro inicial. Se crea de manera predeterminada junto con formularios vacíos de criterios, antecedentes, ficha médica, muerte cerebral y activación familiar.
    \item \textbf{Activo}: El donante potencial cumple con los criterios iniciales.
    \item \textbf{Confirmado}: Tras completar y validar todos los formularios obligatorios, la sospecha es confirmada por el coordinador de trasplantes.
    \item \textbf{Alocación}: Se han registrado órganos disponibles y se han abierto ofertas a otros hospitales.
    \item \textbf{Cirugía}: Se ha programado la cirugía de extracción en pabellón.
    \item \textbf{Trasplantado}: Cierre del ciclo de procuramiento (cuando todos los órganos de la sospecha han sido trasplantados o desestimados).
    \item \textbf{Desestimado / Descartado}: El caso es cancelado o rechazado debido a razones médicas o por denegación familiar.
\end{itemize}

\subsection{2. Estados de un Órgano (Organ)}
\begin{itemize}
    \item \textbf{Pendiente}: Registrado pero sin ofertas activas.
    \item \textbf{Alocando}: Ofertas activas en curso hacia hospitales receptores.
    \item \textbf{Asignado}: Una oferta fue aceptada por un hospital receptor y asignada por el sistema.
    \item \textbf{InicioProcura}: Comienzo del procedimiento quirúrgico específico para este órgano.
    \item \textbf{CrossClamp / En isquemia}: Clampeo aórtico aplicado; comienza a correr el tiempo de isquemia fría.
    \item \textbf{FinalProcura}: Extracción quirúrgica finalizada.
    \item \textbf{Trasplantado}: Trasplantado exitosamente en el receptor en el hospital destino.
    \item \textbf{Desestimado}: Órgano descartado durante el proceso debido a inviabilidad médica sobrevenida u otros problemas técnicos.
\end{itemize}

\subsection{3. Estados de una Oferta (Offering)}
\begin{itemize}
    \item \textbf{Pendiente}: Esperando respuesta del centro receptor prioritario.
    \item \textbf{Aceptada}: El centro receptor aceptó la oferta del órgano.
    \item \textbf{Rechazada}: El centro receptor rechazó el órgano (lo que activa automáticamente al siguiente centro en la cola de prioridad).
    \item \textbf{Expirada}: El tiempo de espera límite del centro receptor expiró sin una respuesta.
\end{itemize}

\section{Reglas de Negocio Clave}

\begin{reglabox}{Criterios de Confirmación de Sospecha}
Para que una sospecha (Suspicion) pueda ser confirmada (habilitando la oferta de sus órganos), debe cumplir con cinco requisitos obligatorios que deben estar presentes y formalmente confirmados por personal autorizado:
\begin{enumerate}
    \item \textbf{Antecedentes Médicos (Background)}: Confirmado.
    \item \textbf{Criterios de Inclusión (InclusionCriteria)}: Confirmado.
    \item \textbf{Ficha Médica (MedicalRecord)}: Confirmado.
    \item \textbf{Muerte Cerebral (BrainDeath)}: Confirmado.
    \item \textbf{Activación Familiar (FamilyActivation)}: Confirmado (con consentimiento positivo de donación).
\end{enumerate}
El backend expone \texttt{enableToConfirm} que valida que estas 5 relaciones tengan \texttt{confirmation.status === true}.
\end{reglabox}

\vspace{0.3cm}

\begin{reglabox}{Cola de Ofertas y Expiración Automatizada}
Cuando se ofrece un órgano (\texttt{createOffering}), se puede ingresar una lista priorizada de hasta 5 centros médicos (\texttt{medicalCenterIds}):
\begin{itemize}
    \item La oferta número 1 de la lista se activa inmediatamente (\texttt{activeDate = new Date()}).
    \item El backend utiliza un job programado (\texttt{node-schedule}) con una expiración de 1 hora por defecto.
    \item Si el centro médico actual no responde a tiempo, la oferta expira y el sistema activa de forma secuencial la siguiente oferta (prioridad \texttt{priority + 1}) actualizando su \texttt{activeDate}.
    \item Si un centro Acepta, se cancelan (expiran) el resto de las ofertas y el órgano pasa a estar Asignado al hospital que aceptó.
    \item Si todas las ofertas de la cola son Rechazadas o Expiran, el estado del órgano regresa a Pendiente y se eliminan las ofertas activas para permitir una nueva ronda de asignación.
\end{itemize}
\end{reglabox}

\newpage

\begin{conceptobox}{Control del Tiempo de Isquemia Fría}
Cuando se efectúa el clampeo aórtico (\texttt{crossClamp} en \texttt{Surgery}), se propaga este timestamp a todos los órganos asociados:
\begin{itemize}
    \item Cada tipo de órgano (\texttt{OrganType}) posee un límite de isquemia máximo permitido (\texttt{ischemiaTime} en horas; ej. el corazón tiene pocas horas de viabilidad, mientras que los riñones tienen mayor margen).
    \item El sistema calcula el tiempo límite de viabilidad: \texttt{crossClamp + ischemiaTime}.
    \item Utilizando la librería \texttt{countdown}, el panel web muestra un reloj regresivo dinámico formateado en español (ej. ``2 horas, 14 minutos'') indicando el tiempo de isquemia fría restante.
    \item Si el tiempo actual sobrepasa el límite calculado, el estado del temporizador se muestra como ``Tiempo de isquemia finalizado''.
\end{itemize}
\end{conceptobox}

\vspace{0.3cm}

\begin{conceptobox}{Notificaciones Push en Dispositivos Móviles}
El backend integra el SDK de Expo (\texttt{expo-server-sdk}) para enviar alertas push automáticas a coordinadores y procuradores:
\begin{itemize}
    \item \textbf{Creación de sospecha}: Cuando se levanta una sospecha en un hospital, el sistema busca a todos los usuarios con rol \texttt{procurer} asociados a ese hospital (\texttt{Workplace}) y les envía una notificación push instantánea: ``Nueva sospecha levantada. Presione aquí para ver en aplicación''.
    \item \textbf{Actualizaciones de Criterios y Fichas}: Alerta a los coordinadores y médicos de la UCI involucrados para dar seguimiento al estado del donante.
\end{itemize}
\end{conceptobox}

\section{Guía de Instalación y Ejecución Local}

\subsection{1. Prerrequisitos en el Sistema}
Node.js (v18.x), PostgreSQL con usuario/contraseña \texttt{iic2154}/\texttt{iic2154} y bases de datos \texttt{backend\_grupo\_2} y \texttt{backend\_grupo\_2\_test}.

\subsection{2. Configurar y Levantar el Backend}
\begin{enumerate}
    \item Instalar dependencias e inicializar variables en el archivo \texttt{.env}:
\begin{lstlisting}[language=bash]
cd backend
npm install --legacy-peer-deps
\end{lstlisting}
    \item Ejecutar las migraciones y seeders para poblar la base de datos local:
\begin{lstlisting}[language=bash]
npx sequelize-cli db:migrate
npx sequelize-cli db:seed:all
\end{lstlisting}
    \item Iniciar el servidor:
\begin{lstlisting}[language=bash]
npm run dev
\end{lstlisting}
\end{enumerate}

\subsection{3. Configurar y Levantar el Frontend Web}
\begin{enumerate}
    \item Instalar dependencias en \texttt{web/}:
\begin{lstlisting}[language=bash]
cd ../web
npm install --legacy-peer-deps
\end{lstlisting}
    \item Levantar el panel web React:
\begin{lstlisting}[language=bash]
npm start
\end{lstlisting}
\end{enumerate}
